# AutumnRecuitmentSkills

> 给正在准备秋招、暑期实习和日常实习的后来者：这套 skill 合集不是为了把求职变得机械，而是为了让你在信息爆炸里少走一点弯路，把精力放回真正重要的事情上。它覆盖求职的两端——**投递前端**（系统整理岗位 JD、去重分类、匹配简历）和**面试后端**（收集面经、构建业界问题分布、复盘个人项目）。

## 写给学弟学妹的一封说明

如果你正在找实习，应该很快就会发现，最累的地方往往不是“写一份简历”本身，而是每天在不同平台、不同公司官网、不同内推链接之间来回切换：这个岗位是不是还在招？这个 JD 和昨天看到的是不是重复？这个岗位更适合哪一版简历？官网链接、Boss、实习僧、邮箱和微信入口到底哪个才是有效投递渠道？

这些事情单独看都不难，但堆在一起就会消耗大量判断力。尤其到了集中投递阶段，如果没有一套稳定的整理方法，很容易出现三个问题：岗位信息越收集越乱，简历版本和岗位要求对不上，最后投递的时候只能凭印象匆忙决定。

我创建这个仓库，是因为自己在暑期实习找工作的过程中真切感受到：系统性整理信息本身就是求职的一部分。把岗位 JD 完整记录下来，把岗位按照技术栈和简历版本归类，把投递链接和状态维护清楚，并不是形式主义，而是在帮助自己更有目的地投递。你越早建立这套秩序，后面越不容易被海量岗位拖着走。

走到面试这一步，又会遇到同样的问题：网上面经铺天盖地，但哪些问题是业界真的高频在问、哪些只是个例？自己按简历项目准备的逐字稿，是不是在自娱自乐、押了根本不会被问到的题？所以这个仓库的第二个 skill，就是把“收集面经 → 统计问题频率 → 用业界分布反过来核对自己的项目准备”这条链路也沉淀下来。

这两个 skill 的目标，都是把零散的整理动作，沉淀成可复用、可持续喂养的工作方式。

## 当前包含的 Skill

### 1. `skills/recruiting-jd-sync` —— 岗位 JD 信息整理（投递前端）

它用于协助完成招聘岗位信息的系统化整理，包括：

- 从招聘网站、公司官网或收藏列表中收集岗位；
- 基于详情页完整复现 JD 信息；
- 按 URL、岗位 ID、公司、岗位名、城市和 JD 文本去重；
- 根据岗位技术栈和可用简历版本分配 `D` 类入库编号；
- 同步维护 JD 字典 Markdown、岗位分类 Markdown 和投递 Excel；
- 校验 Excel 中的格式、超链接、分组 sheet、投递状态颜色和编号一致性。

### 2. `skills/interview-qa-kb` —— 面试问答知识库维护（面试后端）

它用于把收集到的面经沉淀成“可复习、可复盘”的面试问答知识库，并据此核对个人项目准备，包括：

- 把面经/Q&A 按主题归类、自管编号（前缀-N）、去重合并、标注来源；
- 维护两类母文件：**答案库**（业界标准答案，按章节）+ **问题频率/分布库**（问题 + 频率 + 来源公司 tag + 状态），用频率浮出业界高频考点；
- 按章节把答案库拆成子文件便于消化，以编号为锚做**母↔子双向同步**；
- 对项目题做 **`标准答案 ¦ 我的项目回答`** 双轨；
- 把项目逐字稿与业界问题分布做**交集核对**（通用高频题覆盖度 + 项目追问套路覆盖度），并标出“自娱自乐风险”；
- 缺图/缺公式时**重绘补全**（联网核实），每批做一致性/时效性自检。

两个 skill 互补：先用 `recruiting-jd-sync` 把“投哪些岗、用哪版简历”理清楚，再用 `interview-qa-kb` 把“会被问什么、自己项目怎么答”准备扎实。

## 推荐使用方式

这两个 skill 都只依赖一个 `SKILL.md` + 纯 markdown 文件，**跨平台、与具体工具无关**，可用于 Codex、Claude Code、Cowork 等任何支持 SKILL.md 的环境。

把需要的 skill 复制到本地 skill 目录（以 Codex 为例）：

```
mkdir -p ~/.codex/skills
cp -R skills/recruiting-jd-sync ~/.codex/skills/
cp -R skills/interview-qa-kb   ~/.codex/skills/
```

之后这样触发：

```
# 投递前端：收集去重分类岗位、同步 JD 字典/分类/投递表
Use $recruiting-jd-sync to collect new internship jobs, deduplicate them, classify them by JD technical stack and resume availability, and synchronize the JD dictionary, classification file, and application workbook.

# 面试后端：整理面经进知识库、统计频率、核对项目逐字稿
Use $interview-qa-kb to organize this batch of interview questions into the Q&A knowledge base, dedupe and update question frequency with company tags, and cross-check my project script against the industry question distribution.
```

如果你已经有自己的文件结构，建议保持类似分工：

- **JD 汇总文件**：作为岗位 JD 字典，完整保存官方或招聘页信息；
- **分类文件**：根据技术栈、匹配度和简历版本做分析；
- **Excel 表格**：作为真正投递时使用的操作清单；
- **问答母文件（答案库 + 频率库）**：作为业界问题分布的字典，持续喂养；
- **项目逐字稿**：按项目维护，是真正面试时用的稿子。

## 维护原则

这个仓库会优先维护可复用的流程和检查规则，而不是某一次求职记录本身。具体岗位、公司、简历版本和面经内容可以随个人情况变化，但下面这些原则应该保持稳定：

- 信息源要可追溯；
- JD 描述 / 面经问题要完整；
- 分类与编号依据要来自客观要求（岗位要求、问题主题），不靠拍脑袋；
- 投递链接要准确，问题频率统计要去重；
- Excel 格式和状态标记要可信；
- 缺图/缺公式要如实复原、不臆造；
- 最终目标是帮助“投递决策”和“面试准备”，而不是堆积数量。
